Start here

Agent Builder: intrinsically secure

Agent Builder - also called the Copilot Studio agent builder or the declarative agent builder - is the lightweight, no-code experience embedded directly in Microsoft 365 Copilot for building declarative agents.1 It is reachable from microsoft365.com/chat, office.com/chat, and the Microsoft 365 Copilot app in Teams desktop and web (both the Work and Web toolbars), and is not available on mobile.1

The agents it produces are declarative agents: customized versions of Copilot with their own instructions, knowledge, and conversation starters, running on the same Copilot infrastructure. Microsoft draws a hard boundary with full Copilot Studio - "If you need more advanced capabilities like Actions to integrate external services, use Microsoft Copilot Studio." Those advanced capabilities are a separate product; everything in this guide concerns the declarative agents built in Agent Builder, which are governed entirely through Microsoft 365 admin controls.1

🛡
Built-in guardrails - what Agent Builder agents can't do, by designThese constraints are inherent to the declarative agent model. They hold regardless of how an agent is configured or who it's shared with.
  • No autonomous operation. An agent only acts in direct response to a user's prompt in a chat - it never runs on its own initiative or in the background.
  • No event triggers. Agents cannot be started by an event, a schedule, a webhook, or any external signal. There is no trigger surface to govern.
  • Read-only by nature. They cannot edit, create, or delete existing files or data in any system. They retrieve and generate responses and can create new file artifacts, but they do not make updates to existing data.
  • Admin-allowed data sources only. An agent can connect only to the data sources the Microsoft 365 admin has explicitly enabled - connectors, knowledge sources, and grounding are all gated by tenant policy (see Dimension 5).
  • Always the end user's permissions. Every response is constrained to what the signed-in user is already permitted to access. An agent can never surface data the person running it couldn't reach themselves (see Dimension 2).
  • If you want to build an agent that operates autonomously, is triggered by an event, or writes to external systems, you need full Copilot Studio.

Roles authorized for agent management

Agent management in the M365 admin center is governed by Microsoft Entra admin roles.7

Least-privilege tip

The AI Administrator role gives you the same tenant-wide control over agents as Global Administrator, but without all the other permissions a Global Admin carries. Microsoft recommends using it instead wherever you can.7

1Dimension 1Controlling who can build agents
Usually not worth turning off

There's rarely a good reason to turn Agent Builder off. The agents it creates are safe by design: they can't create, update, or delete data, and they only ever return what the signed-in user can already see (see the guardrails at the top of this guide). Because the risk is so low, most tenants are better off limiting who can build - scoping to Specific users / groups - than switching it off entirely.

Default state: Agent Builder is on out of the box - "the capability is enabled by default in all Microsoft 365 Copilot licensed tenants."5

The primary control - User access

A single setting turns the Agents -> Agent Builder entry point on or off and scopes it to specific people. Microsoft maps it explicitly: the control "Allow the following users access to Copilot agents" performs the scenario "Enable or disable the entry point for Agent Builder in Microsoft 365 Copilot (Create an agent)."2

Microsoft 365 admin center→Agents→Settings→User access

The setting has three options:6

  • All users (default) - everyone with a Copilot license can build in Agent Builder.
  • No users - the Agent Builder entry point is removed tenant-wide.
  • Specific users / groups - Agent Builder is available only to chosen users or security groups.
Before you turn it off, know what else goes with it

If your admin team really does want to remove the capability, the steps above work - just know it takes more than the Create an agent button with it:

  • Other custom-built agents in your tenant also disappear from users' experience.
  • Users will no longer see Microsoft's own agents - Researcher, Analyst, and the Office 365 agents - in the UI, even though they retain the right to use them.

So it's worth thinking through: turning Agent Builder off changes a lot more than just hiding the build button.

Restricting which agent types appear

A complementary control limits which categories of agents users can view/install:

Microsoft 365 admin center→Agents→Settings→Allowed Agent Types

If you disable a type, those agents don't appear in the Agent store. One nuance: agents built by Microsoft remain visible even when the setting is disabled - but users won't be able to install or use them.6

Roles required: Global Administrator or AI Administrator.7

2Dimension 2How agents get shared - and how admins govern it

The three sharing scopes a maker sees

After a user clicks the Create button on their Agent, "a confirmation message indicates that the agent is private and only available to you."3 They can then choose:

ScopeWho can use it
Only you (default)Only the author can use the agent.3
Specific users in your organizationNamed individuals, security groups, security-enabled M365 groups, or teams via a sharing link. Sharing to a team can post a discovery notification to its home channel.3
Anyone in your organizationAny user in the tenant can use the sharing link.3

Sharing is link-based and grants limited direct access - "sharing an agent doesn't allow for deployment across the organization or integration with other channels." Broad, multi-channel distribution requires Copilot Studio.3

Admin governance of sharing

M365 admin center→Agents→Settings→Sharing

Tenant admins constrain sharing org-wide with the same three-way choice (All users / No users / Specific users). Scope matters: "Only agents built with Microsoft 365 Copilot Agent Builder are governed by sharing control."6 Note that No users disables org-wide sharing but "users can still share directly with specific individuals."6

Keep in mind

Sharing changes aren't retroactive. "Changes to admin controls apply only to new sharing actions. Existing shared agents remain accessible unless you manually update the sharing option." When org-wide sharing is disabled, the Anyone in your organization option greys out, and existing owners hit a "Can no longer update agent" error until they pick a compliant scope.3

Publishing to the org catalog (approval flow)

To make an agent broadly discoverable, a maker can submit it to the org catalog: "An admin reviews the submission in the Microsoft 365 admin center and, if approved, publishes the agent in the Built by your org section of the Agent Store." The shared version and the store version are managed separately.3

SharePoint knowledge respects the end user's permissions

When sharing to specific users, SharePoint sources can be auto-shared, but "the agent respects the end user's information and sensitivity privileges" - content a recipient can't access is excluded from responses, and "file sensitivity labels applied to files in SharePoint or OneDrive are respected." Auto-sharing of SharePoint files/folders is supported only when sharing with specific security groups, not with everyone.31

3Dimension 3Tracking & managing agents across the tenant

Finding and managing your Agent Builder agents in the registry

M365 admin center→Agents→All Agents→Registry

The registry is the inventory for every agent in your tenant, including the ones people build with Agent Builder.9 To work with just those, apply the Platform filter and set it to Agent Builder - the list narrows to your Agent Builder agents, each showing its name, creator, creation date, host products, and availability status.95

Click any agent in that filtered list to open its detail view, where you can review its knowledge source details, agent instructions, sharing configuration, and more, so you can see exactly what an agent can reach and who it's shared with. From the list you can also Export to CSV to keep an offline inventory.9

While you're there, the summary tiles flag Agents without owners and Unmanaged agents, which point you to the ones that need a new owner or a closer look.9

Requests tab. Agents that makers submit to your organization's catalog appear in the Requests tab of this screen, where you review them and approve or reject before they're published to the Built by your org store (see Dimension 2).3

ActionEffect

Agent Management Rules - bulk lifecycle 2026

MAC now offers rules-based bulk governance to "apply governance and lifecycle controls across AI agents at scale." Two scenarios are documented:6

ScenarioEffect
Install Microsoft AgentsBulk-installs Microsoft first-party agents across the tenant.6
Reassign ownerless agents created with Agent Builder to managerAgent-Builder-specific - transfers ownership "to the manager of the previous owner based on Microsoft Entra ID hierarchy."6
Ownership transfer is an admin-only path

Microsoft 365 Copilot "doesn't support transferring ownership of declarative agents" through the maker UI - admin reassignment via Agent Management Rules is the route.3

Usage & reporting

After selecting an agent in the Registry list, you can view usage specific to the selected agent.

Broad usage and adoption signals organization-wide appear in Purview DSPM for AI reports and the Agent overview workload, which provides usage and insights that help monitor agent adoption and governance. If Agent 365 is enabled, additional security detection and reporting is available in Purview as well.810

Microsoft Agent 365

Microsoft Agent 365 is the control plane for AI agents and adds additional tools and capability to confidently deploy, govern, and manage all your agents at scale, regardless of where these agents are built or acquired.59

4Dimension 4Identifying risky agents

As noted above, Agent Builder agents are inherently low risk due to capability restrictions and intrinsically secure by virtue of inheriting existing M365 permissions to SharePoint, OneDrive and other knowledge sources. However, there are security tools available for proactive and reactive AI agent management.

Data risk can be identified and managed through two stacks: Microsoft Purview (data security & compliance) and SharePoint Advanced Management (oversharing).

Microsoft Purview

Purview supports the full compliance toolset against Copilot/agent interactions: DSPM for AI, auditing, data classification, sensitivity labels, DLP, Insider Risk Management, communication compliance, eDiscovery, data lifecycle management, and Compliance Manager.10

Purview portal→Solutions→DSPM for AI

  • DSPM for AI is the best place to start - "discover, secure, and apply compliance controls for AI usage … Data risk assessments help you identify and fix issues that could result in oversharing of data." One-click policies include Detect risky interactions in AI apps, Detect unethical behavior in AI, and Protect items with sensitivity labels from Microsoft 365 Copilot and agent processing. Reports include Potential risky AI usage and Top unethical AI interactions.10
  • Audit: agent prompts and responses are "captured in the unified audit log."10
  • DLP for Copilot: the "Microsoft 365 Copilot and Copilot Chat" policy location can "restrict the processing of prompts that contain sensitive information types, or processing files and emails that have specific sensitivity labels."10
  • Insider Risk Management: the Risky AI usage template detects "prompt injection attacks and accessing protected materials."10
  • eDiscovery / retention: queryable via the Copilot activity condition; retained via the Microsoft Copilot Experiences location.10

SharePoint Advanced Management - oversharing controls

  • Restricted Content Discovery (RCD): when enabled for a site, "content from that site doesn't appear in organization-wide search and Microsoft 365 Copilot experiences unless a user recently interacted with the content" - without changing existing permissions. It also "restricts usage of AI-powered features in SharePoint … Users don't see entry points such as the Copilot button, AI actions menus (including creating agents)", so on sensitive sites RCD also stops people from building agents there.11

    SharePoint admin center→Sites→Active sites→Settings→Restrict content from Microsoft 365 Copilot

  • Restricted SharePoint Search interacts directly with Agent Builder: "If Restricted SharePoint Search is enabled, you can't use SharePoint as a knowledge source."4
  • Supporting tools: SharePoint Admin Agent, content management assessment, Data Access Governance reports, and catalog management help locate oversharing-prone sites before a Copilot rollout.12

Graph / Copilot connectors

"Admins must enable and configure Copilot connectors in the Microsoft 365 admin center." Depending on the control, connectors are managed from the Microsoft Entra admin center and the Microsoft 365 admin center - including app registration / admin consent in Entra and enable/disable in MAC. If a scoped attribute doesn't appear, the documented cause is that "your administrator didn't configure the scoped content."24

Web & SharePoint source controls

  • Web grounding is governed by the Allow web search in Copilot policy, which applies tenant-wide and overrides the maker-facing Web content toggle.1
  • SharePoint as a source is disabled wholesale when Restricted SharePoint Search is on.4
  • Hard limit: "Agent Builder … doesn't support blocking general AI knowledge from your agent's responses. For stricter control over knowledge sources, use Copilot Studio."4

Where source control actually lives

For Agent Builder agents, the available data sources are governed entirely through Microsoft 365 admin controls. The three levers are:

  • Microsoft 365 admin center connector enablement - a knowledge source (such as a Graph / Copilot connector) is selectable only if an admin has enabled it; if it isn't configured, it doesn't appear to makers.24
  • The "Allow web search in Copilot" policy - controls web grounding tenant-wide and overrides the maker-facing Web content toggle.1
  • Restricted SharePoint Search - when enabled, SharePoint can't be used as a knowledge source at all.4
An agent can never out-reach its user

Beyond admin enablement, every source an agent draws on is still bounded by the signed-in user's own permissions - an Agent Builder agent cannot surface data the person running it isn't already entitled to see (see the guardrails at the top and Dimension 2).3

6Dimension 6Controlling who can use agents

Audience is the intersection of two gates: the agent must be shared with the user and the user must fall within the org's User access scope. Capabilities gate further by license: "Users can only add agents that are shared with them when they have the type of Microsoft 365 Copilot license required … attempts to use the agent might result in an error."3

Licensed users - included, no extra setup

For Microsoft 365 Copilot licensed users - the primary Agent Builder audience - there's nothing to provision and no consumption based cost - usage of Agent Builder agents comes with Microsoft 365 Copilot licenses. Agents built in Agent Builder are included also do not consume tenant Dataverse storage.121

Copilot Chat (free tier) - what's free vs. gated

For users on Copilot Chat (the free / unlicensed non-M365 tier), declarative agents that are grounded in instructions and use only public websites are available at no additional cost.12
